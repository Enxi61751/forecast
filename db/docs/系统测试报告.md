## 二、发现并修复的问题

### 问题 1：模型服务 Python 版本兼容性（`features.py`）

**现象：** 模型服务启动时报错 `TypeError: unsupported operand type(s) for |: 'type' and 'NoneType'`

**原因：** `features.py` 使用了 Python 3.10+ 的联合类型语法 `np.ndarray | None`，但运行环境是 Python 3.8。

**修复：** 在 `features.py` 首行添加 `from __future__ import annotations`。

---

### 问题 2：Pydantic v2 与 Python 3.8 不兼容

**现象：** 模型服务启动时报 `TypeError: Unable to evaluate type annotation 'list[TimePoint]'`

**原因：** Pydantic v2.10.0 版本起**正式放弃 Python 3.8 支持**（参见 Pydantic changelog）。安装的 2.10.6 版本会将 `List[T]` 在内部规范化为 `list[T]`（Python 3.9+ 写法），在 Python 3.8 上求值时失败。

**修复：**
1. 将 `schemas.py` 中所有 `list[T]` / `dict[K, V]` 改为 `List[T]` / `Dict[K, V]`（来自 `typing` 模块）
2. 移除 `schemas.py` 中的 `from __future__ import annotations`（该声明会让 Pydantic v2 将类型注解转成字符串再重新求值，触发内部规范化）
3. **降级 pydantic 至 2.9.2**（最后一个支持 Python 3.8 的版本）：
   ```bash
   pip3 install "pydantic==2.9.2"
   ```

---

### 问题 3：后端与模型服务的请求结构不匹配

**现象：** 后端调用模型 `/predict` 时返回 `422 Unprocessable Entity: Field required: series`

**原因：** 后端 `PredictRequest` 使用通用 `payload: Map<String, Object>` 字段嵌套 `series/events`，而模型期望这些字段在请求体顶层。`ModelServiceClient.predict()` 直接转发了后端 DTO，结构不一致。

**修复：** 修改 `ModelServiceClient.predict()` 方法，将 `payload` 内容展开并与 `target/horizon/asOf` 合并后再发送给模型：

```java
// 修改前
public PredictResponse predict(PredictRequest req) {
    return restClient.post()
            .uri(baseUrl + "/predict")
            .body(req)   // 直接转发，结构不匹配
            .retrieve()
            .body(PredictResponse.class);
}

// 修改后
public PredictResponse predict(PredictRequest req) {
    Map<String, Object> modelReq = new HashMap<>(req.getPayload()); // 展开 payload
    modelReq.put("target", req.getTarget());
    modelReq.put("horizon", req.getHorizon());
    if (req.getAsOf() != null) {
        modelReq.put("asOf", req.getAsOf().toString());
    }
    return restClient.post()
            .uri(baseUrl + "/predict")
            .body(modelReq)  // 发送展开后的结构
            .retrieve()
            .body(PredictResponse.class);
}
```

## 三、测试结果

### 3.1 模型服务接口

| 接口 | 方法 | 测试结果 |
|---|---|---|
| `/health` | GET | ✅ `{"ok": true}` |
| `/predict` | POST | ✅ 返回预测结果（stub 模式，lgbm=73.1, tft=73.1，极端风险=MID_RISK） |

**`/predict` 测试请求（使用数据库中真实数据构造）：**
- 价格序列：2026-03-01 至 2026-03-10，WTI 68.5 ~ 73.1

```json
{

  "extremeClass": {"label": "MID_RISK", "prob": 0.5},

    "lgbm_top_features": ["last_price", "ret1", "sent_last", "evt_cnt_7d", "evt_max_intensity_7d"]

}
```

### 3.3 端对端链路验证

```
客户端 → POST /api/predict/run（后端）
           ↓ ModelServiceClient.predict()（展开 payload）
         POST /predict（模型服务）
           ↓ PredictPipeline.run()（CEEMDAN + LGBM + TFT stub）
         返回 PredictResponse
           ↓ PredictionRunRepo.save()（持久化到 MySQL）
           ↓ Redis 缓存（TTL=5分钟）
         返回给客户端
```

```sql
SELECT id, target, horizon, run_at FROM prediction_run ORDER BY id DESC LIMIT 2;
-- 2  WTI  1d  2026-03-17 03:37:36

```

两次预测结果均成功持久化。

## 四、启动顺序

```bash
# 1. 启动 Redis
redis-server --daemonize yes

# 2. 启动模型服务（model-remote 分支目录）
cd /tmp/model-service
USE_STUB=1 python3 -m uvicorn app.main:app --host 0.0.0.0 --port 8000 &

# 3. 启动后端（yyy-dev 分支目录）
cd ~/forecast
mvn spring-boot:run -Dspring-boot.run.profiles=local
```

> **注意：** curl 测试时需设置 `no_proxy=localhost`，否则系统代理（172.29.0.1:7897）会拦截本地请求。
