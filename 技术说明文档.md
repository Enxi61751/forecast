## 0. 系统架构概览

> 本节面向非技术背景的读者，从整体视角阐述系统的定位、组成与核心业务流程，旨在帮助评审人员在深入阅读技术细节之前建立完整的系统认知。

### 系统定位

本系统是一套面向大宗商品交易场景的**智能决策支持平台**，面向银行交易员与风控人员提供端到端的油价预测与风险分析服务。系统以多源数据为输入（实时行情、宏观指标、新闻舆情），经由机器学习预测、多智能体市场模拟和大语言模型报告生成三个核心环节，最终输出结构化预测结论与可供直接阅读的自然语言风险报告。此外，系统支持对多种量化交易策略进行历史回测，为策略有效性评估提供量化依据。

### 核心业务流程

| 阶段 | 说明 |
|------|------|
| **① 行情数据采集** | 系统定时（每日凌晨 1 时）自动拉取 WTI/Brent 原油的 OHLCV 行情数据，亦支持按需触发；历史数据统一存储于 MySQL |
| **② 新闻舆情分析** | 新闻稿件经 ML 服务的 NLP 模型处理，输出情感评分（区间 \[-1, +1\]，正向偏多、负向偏空）及极端事件标注（如 OPEC 减产、地缘冲突等） |
| **③ 多模型价格预测** | 业务层将历史行情与技术指标作为特征，送入 ML 服务，经梯度提升集成、时序深度学习（TFT）与动态权重融合（MWUM）三模型协同推断，输出带置信区间的多步价格预测路径 |
| **④ 多智能体市场模拟** | Agent 服务基于 stagex2 框架，以价格预测结果为触发事件，模拟 CTA 趋势追随者、套期保值者等典型市场参与者的决策行为，输出结构化推理材料 |
| **⑤ 风险报告生成** | 将量化预测结论与智能体推理材料作为上下文，调用 DeepSeek Reasoner 大语言模型，自动生成面向业务人员的结构化自然语言风险报告 |
| **⑥ 策略历史回测** | 基于真实历史 OHLCV 数据，对 Aberration、DualThrust、MACD、Momentum 等量化策略进行全面回测，支持与 ML 预测信号组合，输出净值曲线、最大回撤、样本外指标等绩效统计 |

## 1. 项目概述

本系统是面向金融机构的**原油价格智能预测与风险分析平台**后端服务，旨在为银行大宗商品交易团队提供数据驱动的油价预测、舆情感知和风险评估能力。系统整合了多源数据（实时油价、宏观经济指标、新闻舆情、地缘政治事件），借助机器学习预测模型与多智能体博弈模拟，输出结构化预测结果和可读性强的风险报告，辅助交易员和风控人员的决策制定。

**核心能力：**

- **实时油价采集** —— 自动拉取 WTI/Brent 原油实时及历史行情（OHLCV）
- **新闻舆情分析** —— 接入新闻稿件，调用 NLP 模型进行情感评分与极端事件检测
- **多模型融合预测** —— 融合 AlphaModelFrame 集成学习、TFT（Temporal Fusion Transformer）深度学习、MWUM 动态加权三类模型，输出多时间尺度价格预测
- **多智能体风险模拟** —— 基于 stagex2 框架，模拟 CTA、套期保值者等不同市场参与者的博弈行为
- **LLM 报告生成** —— 调用 DeepSeek 大语言模型，将量化分析材料转化为面向业务人员的自然语言风险报告

## 3. 系统架构设计

本系统采用**经典分层架构**，并通过微服务解耦将机器学习推理与业务逻辑分离：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         前端 (Vue 3 + Vite)                         │
│                        http://localhost:5173                        │
└──────────────────────────────┬──────────────────────────────────────┘
                               │ HTTP / JSON
                               ▼
┌─────────────────────────────────────────────────────────────────────┐
│                    Spring Boot 后端 (port 8080)                     │
│                                                                     │
│  ┌───────────┐   ┌───────────┐   ┌───────────┐   ┌──────────────┐ │
│  │ Controller │──▶│  Service  │──▶│ Repository │──▶│  MySQL DB    │ │
│  │   Layer    │   │   Layer   │   │   Layer    │   │              │ │
│  └───────────┘   └─────┬─────┘   └────────────┘   └──────────────┘ │
│                        │                                            │
│                        │ RestClient                                 │
│                        ▼                                            │
│               ┌────────────────┐                                    │
│               │ ModelService   │─── Redis (热缓存)                   │
│               │ Client         │                                    │
│               └───┬────┬───┬──┘                                     │
└───────────────────┼────┼───┼────────────────────────────────────────┘
                    │    │   │
         ┌──────────┘    │   └──────────┐
         ▼               ▼              ▼
┌────────────────┐ ┌──────────┐ ┌──────────────────┐
│ ML Model 服务   │ │ Agent    │ │ DeepSeek LLM API │
│ FastAPI :8000  │ │ 服务     │ │ (云端)            │
│ ┌────────────┐ │ │ :8001    │ │                   │
│ │ Alpha 集成  │ │ │ stagex2  │ │ deepseek-reasoner │
│ │ TFT 时序   │ │ │ 多智能体  │ │                   │
│ │ MWUM 融合  │ │ │ 博弈模拟  │ │                   │
│ └────────────┘ │ └──────────┘ └──────────────────┘
└────────────────┘
```

**架构要点：**

1. **关注点分离** —— Controller 仅负责请求路由与参数校验，Service 封装业务逻辑，Repository 处理数据持久化
2. **模型服务解耦** —— ML 推理部署为独立的 Python FastAPI 服务，通过 HTTP/JSON 与 Java 后端通信，避免语言绑定
3. **弹性降级** —— 每个外部服务调用均有 try-catch 包裹，失败时自动回退到 Mock 数据，保证系统可用性
4. **缓存加速** —— Redis 缓存最新预测结果，减少重复计算开销

## 6. 核心功能模块详解

### 6.1 原油价格数据采集

**涉及组件：** `OilPriceController` → `OilPriceService` → `OilPriceHistoryRepo` / 外部 API

**实现原理：**

系统通过两种途径获取油价数据：

1. **实时拉取（按需）** —— 前端调用 `GET /api/oilprice/latest` 时，`OilPriceService` 通过 `RestTemplate` 向 OilPriceAPI（`https://api.oilpriceapi.com/v1/prices/latest`）发起带 Bearer Token 的 HTTP GET 请求，获取 WTI 和 Brent 两个品种的最新价格，返回标准化的 `OilPriceApiResponse` 给前端。

2. **定时采集（每日）** —— `DailyPipelineJob` 以 cron 表达式 `0 0 1 * * ?`（每日凌晨 1:00）触发，调用 `OilPriceService.fetchAndSaveOilPrice()`，将 WTI 和 Brent 的最新价格分别持久化为 `OilPriceHistory` 实体存入数据库。价格时间戳通过解析 API 返回的 ISO 8601 `OffsetDateTime` 并转换为 `LocalDateTime` 实现时区对齐。

3. **历史查询** —— `GET /api/oilprice/history?start=...&end=...` 支持按时间范围查询数据库中的油价历史记录，利用 JPA 的 `findByTimestampBetween` 自动生成 SQL Range 查询。

**容错设计：** 若 API Token 未配置（保持默认值 `YOUR_OIL_API_TOKEN`）或 API 调用失败，自动返回 Mock 数据，避免服务不可用。

---

### 6.2 新闻舆情分析与情感评分

**涉及组件：** `NewsController` → `NewsService` → `ModelServiceClient` → ML 模型服务（port 8000）

**实现原理：**

新闻舆情分析采用**采集-评分-存储**三阶段流水线设计：

```
POST /api/news/ingest
        │
        ▼
┌─────────────────────────┐
│ Step 1: 持久化原始稿件    │  @Transactional
│ NewsArticle → MySQL     │  ← 独立事务，确保稿件不丢失
└────────────┬────────────┘
             │ newsId
             ▼
┌─────────────────────────┐
│ Step 2: 调用 NLP 模型    │  HTTP POST → :8000/sentiment/score
│ 情感评分 + 极端事件检测   │  ← 事务外执行，失败不影响 Step 1
└────────────┬────────────┘
             │ SentimentResponse
             ▼
┌─────────────────────────┐
│ Step 3: 持久化评分结果    │  @Transactional
│ SentimentScore + Event  │  ← 评分与事件原子写入
└─────────────────────────┘
```

**关键设计决策：**

- **事务边界精细控制** —— `saveArticle()` 和 `saveSentiment()` 各自拥有独立的 `@Transactional` 注解。模型调用（Step 2）位于两个事务之间，确保即使 NLP 服务不可用，原始新闻稿件依然完整保存。
- **极端事件检测** —— 模型服务返回的 `SentimentResponse` 中可包含 `extremeEvents` 列表（如地缘政治冲击 GeoPoliticalShock、OPEC 减产 SupplyCut 等），每个事件带有类型、摘要、强度和发生时间，系统逐一写入 `ExtremeEvent` 表。
- **异步降级** —— 情感评分失败仅记录警告日志，不影响新闻入库的返回，前端可后续重试评分。

### 6.3 多模型融合预测引擎

**涉及组件：** `PredictController` → `PredictService` → `ModelServiceClient` → ML 模型服务（port 8000）→ Redis

**实现原理：**

预测引擎是系统的核心功能，整合了三类机器学习模型实现多时间尺度的油价预测：

#### 模型架构

| 模型 | 类型 | 职责 |
|------|------|------|
| **AlphaModelFrame** | 集成学习（pkl 模型文件） | 基于 175 维特征数据集的收益率预测，使用预训练的集成模型或预计算的预测结果 |
| **TFT (Temporal Fusion Transformer)** | 深度学习时序模型 | 油价时序预测，输出带置信区间的价格路径，同时提供极端分类能力 |
| **MWUM (Multiplicative Weights Update Method)** | 动态权重融合 | 基于各子模型历史表现动态调整融合权重，自适应市场状态变化 |

#### 预测请求处理流程

```
前端 POST /api/predict
  Body: { target: "WTI", horizon: "7d", payload: {...} }
         │
         ▼
┌─────────────────────────────────────────┐
│ ModelServiceClient.predict()            │
│ 将 payload 展平，与 target/horizon 合并 │
│ POST → :8000/predict                    │
└────────────────┬────────────────────────┘
                 │ PredictResponse { forecast, extremeClass, explain }
                 ▼
┌─────────────────────────────────────────┐
│ 持久化到 prediction_run 表              │
│ forecast → forecastJson (JSON TEXT)     │
│ extremeClass → extremeClsJson          │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ Redis 缓存                              │
│ Key: pred:latest:{target}:{horizon}     │
│ Value: predictionRunId                  │
│ TTL: 5 分钟                             │
└─────────────────────────────────────────┘
```

**payload 灵活设计** —— `PredictRequest.payload` 采用 `Map<String, Object>` 类型，允许前端传入任意结构的附加参数（如特征数据、时间窗口约束等），`ModelServiceClient` 在发送前将 payload 与 target/horizon 合并为扁平化请求体，适配 Python 模型服务的接口格式。

**预测结果结构** —— `PredictResponse` 包含三个维度：
- `forecast`：价格预测结果（可为路径、区间或点预测）
- `extremeClass`：极端行情分类结果（如 normal/spike/crash 及置信度）
- `explain`：模型可解释性信息（特征重要性等）

### 6.4 多智能体模拟与风险报告生成

**涉及组件：** `ReportController` → `ReportService` → `ModelServiceClient` → Agent 服务（port 8001）+ DeepSeek LLM API

**实现原理：**

风险报告生成是系统最具创新性的功能，采用**"多智能体博弈模拟 + 大语言模型文本生成"**的双阶段架构：

#### 阶段一：多智能体博弈模拟（stagex2 框架）

系统基于 stagex2 框架构建市场微观结构模拟环境，将不同类型的市场参与者建模为智能体，模拟其面对油价预测信号时的行为决策：

```json
{
  "input": {
    "environment": {
      "factor_snapshot": {           // 因子快照
        "trend_score": 0.0,          // 趋势得分
        "rsi_status": "Neutral",     // RSI 状态
        "term_structure": "Contango" // 期限结构
      },
      "tail_risk_report": {          // 尾部风险报告
        "gamma_profile": "Neutral",  // Gamma 暴露
        "liquidity_stress": 0.3      // 流动性压力
      },
      "market_microstructure": {     // 市场微观结构
        "bid_ask_spread": 0.05,      // 买卖价差
        "order_book_depth": "Normal" // 订单簿深度
      },
      "session_info": {              // 交易时段信息
        "phase": "Mid-Day",
        "time_to_close": 240         // 距收盘分钟数
      }
    },
    "event": {                       // 触发事件（来自预测结果）
      "headline": "Oil price forecast: WTI over 7d",
      "body": "<forecastJson>",
      "source": "CitiCup Model",
      "impact_type": "price",
      "sentiment_score": 0.0
    },
    "self": {                        // 默认 CTA 智能体状态
      "role": "CTA",
      "mandate": "trend-following momentum strategy",
      "max_leverage": 3.0,
      "stop_loss_pct": 0.05,
      "position": 0.0,
      "cash_level": 1000000.0
    }
  },
  "current_price": 80.0,
  "current_volatility": 0.2,
  "dealer_inventory": 100
}
```

Agent 服务返回各智能体的决策推理材料（`materialsJson`），为后续报告撰写提供量化论据。

#### 阶段二：LLM 报告生成

系统将智能体模拟产出的结构化材料作为上下文，通过 `ModelServiceClient.callLlm()` 调用 DeepSeek Reasoner 大语言模型（`deepseek-reasoner`），生成面向银行业务团队的自然语言风险报告。

LLM 调用采用标准 OpenAI 兼容接口：
- 请求格式：`{"model": "deepseek-reasoner", "messages": [{"role": "user", "content": prompt}]}`
- 认证方式：`Authorization: Bearer <api-key>`
- 响应解析：提取 `choices[0].message.content`

#### Upsert 语义保证

每个 `PredictionRun` 仅对应一份 `RiskReport`（数据库唯一约束 `unique_prediction`）。若对同一预测重复生成报告，系统查找已有记录并执行更新（而非插入），避免数据冗余。

### 6.5 策略回测模块

**涉及组件：** `BacktestController` → `BacktestService` → `OilPriceDailyOhlcvRepo` / `PredictionRunRepo`

**功能定位：** 基于历史真实 OHLCV 数据，对四种技术交易策略进行全量历史回测，并支持与 ML 预测信号组合叠加，量化评估各策略在样本内及样本外的历史有效性。

#### 支持的策略与组合模式

| 维度 | 可选项 | 说明 |
|------|--------|------|
| **技术策略** | Aberration | 布林带突破（窗口 35，±1σ 进出场） |
| | DualThrust | 双推力通道（窗口 20，k=0.5） |
| | MACD | 标准 MACD（12/26/9） |
| | Momentum | 动量对比（短周期 3 日 vs 长周期 10 日） |
| **组合模式** | baseOnly | 纯技术策略信号，不使用 ML 预测 |
| | AND | 技术信号与 ML 方向信号**同向时**才入场（取交集） |
| | VETO | **ML 信号主导**，技术策略反向时才否决不入场 |
| | ModelOnly | 仅凭 ML 预测方向信号交易（`strategies` 字段传入） |

#### 防止未来数据泄漏

所有技术指标信号均在 **t 日收盘后计算**，再通过 `shiftByOne()` 将信号推移一格，使 **t 日信号在 t+1 日开盘时执行**，消除前向信息泄漏。ML 信号使用当日开盘价与预测收盘价的差值判断方向，符合"模型在开盘前已有预测"的合理假设。

#### 回测执行流程

```
POST /api/backtest/run
  { symbol, startDate, endDate, strategies[], combineModes[], feeSlipRatePerSide }
          │
          ▼
┌─────────────────────────────────────────┐
│ 1. 从 DB 加载 OHLCV（date/open/high/    │
│    low/close 数组）                      │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 2. 加载 ML 预测（按 symbol→target 映射  │
│    查 prediction_run，解析 forecastJson）│
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 3. 计算技术指标方向信号                  │
│    Aberration / DualThrust / MACD /     │
│    Momentum → baseDir[]                 │
│    → shiftByOne() → baseDirForTrade[]   │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 4. 组合信号：baseOnly / AND / VETO      │
│    → finalDir[]（每日 +1/-1/0）          │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ 5. runSingle()：逐日模拟                 │
│    dailyRet = (close-open)/open × dir  │
│    扣除手续费（2 × feeRate per trade）   │
│    累积净值曲线 equity[]                 │
└────────────────┬────────────────────────┘
                 │
                 ▼
              指标输出：胜率、总收益率、最大回撤、
              样本外（后20%）指标、63日滚动回撤、
              成本敏感性（1×/2×/3× 费率）
```

#### 输出指标说明

| 指标 | 说明 |
|------|------|
| `nTrades` | 总交易笔数 |
| `winRate` | 盈利交易占比 |
| `totalReturn` | 全期累计净收益率（equity[-1] - 1） |
| `maxDrawdown` | 全期最大回撤（负数） |
| `oosReturn` | 样本外（后 20%）净收益率 |
| `oosWinRate` | 样本外胜率 |
| `oosMaxDrawdown` | 样本外最大回撤 |
| `worstRollingDrawdown63d` | 63 日滚动窗口内最大回撤（近似季度级别极端风险情景） |
| `equityCurve` | 每日净值列表 `[{date, equity}]` |
| `costSensitivity` | 不同费率倍数下的净收益率与回撤对比 |

#### 数据状态（截至当前）

| Symbol | 数据量 | 日期范围 | ML 预测 |
|--------|--------|---------|---------|
| CL0（WTI 近月） | 5211 条 | 2006-01-03 ~ 2026-02-20 | ✅ 960 条（2022-03 ~ 2025-12，target="WTI"） |
| CL1/CL2/CL3/CL6 | 5128~5167 条 | 同上 | ✅（同 CL0，符号均映射 target="WTI"） |
| LCO1（Brent） | 5386 条 | 2006-01-03 ~ 2026-02-20 | ❌ 无历史预测数据 |

> `symbol` 以 `"CL"` 开头时自动映射 `target="WTI"`，否则映射 `target="BRENT"`。ML 组合策略（AND/VETO/ModelOnly）建议使用 `symbol="CL0"`，以确保与已导入的 WTI 预测数据正确对齐。

#### 历史预测批量导入

`POST /api/backtest/import-predictions` 接受 `{target, horizon, predictions:[{date, value}]}` 格式，将每条记录以 `runAt = 目标日期 UTC 零时` 写入 `prediction_run` 表，并在日期维度进行幂等去重，是向系统补充历史 ML 预测数据的标准入口。

## 8. 缓存机制

系统以 Redis 作为预测结果热缓存层，对同一标的与预测时间窗口的重复请求直接命中缓存，避免触发高开销的模型推理调用。

**缓存策略：**

| 属性 | 值 |
|------|------|
| 存储介质 | Redis（localhost:6379） |
| 客户端 | Spring `StringRedisTemplate` |
| Key 格式 | `pred:latest:{target}:{horizon}`（如 `pred:latest:WTI:7d`） |
| Value | `PredictionRun` 主键 ID |
| TTL | 5 分钟（`Duration.ofMinutes(5)`） |
| 失败处理 | 缓存写入失败仅记录警告日志，不影响预测结果返回 |

**设计考量：** 缓存值仅存储 `PredictionRun` 主键 ID，而非完整预测 JSON，以避免大体量序列化对象过度占用 Redis 内存；完整预测内容按需通过 ID 从 MySQL 检索。

## 9. 定时任务与数据管道

### 9.1 每日油价采集任务

```java
@Scheduled(cron = "0 0 1 * * ?")  // 每日 01:00 执行
public void fetchOilPrice() {
    oilPriceService.fetchAndSaveOilPrice();
}
```

`DailyPipelineJob` 利用 Spring 的 `@Scheduled` 注解实现 cron 定时触发，配合主类上的 `@EnableScheduling` 注解生效。任务执行时调用外部油价 API 并将 WTI/Brent 两个品种的最新价格分别写入数据库。

### 9.2 应用启动数据初始化

`DataInitializer` 实现 `CommandLineRunner` 接口，在 Spring Boot 完成上下文初始化后自动执行：

1. 检查数据库中是否已存在新闻数据
2. 若为空库，初始化基础演示数据：3 条新闻、6 条汇率、16 条油价历史（8 天 × 2 品种）、3 次预测记录、1 份风险报告
3. 独立检查并初始化事件数据（3 条地缘/经济事件）

该机制确保系统在首次部署或数据库重建后即可进入可演示状态。

## 12. 容错与降级策略

系统在多个层面实现了弹性容错设计，确保外部服务故障不会导致整体不可用：

| 组件 | 容错场景 | 降级策略 |
|------|---------|---------|
| `OilPriceService` | API Token 未配置 / API 调用失败 | 返回 Mock 油价数据（WTI $75.50, Brent $80.25） |
| `NewsService` | NLP 模型服务不可达 | 新闻稿件正常入库，情感评分跳过，记录警告日志 |
| `PredictService` | 预测模型服务调用失败 | 返回 Mock 预测结果（含默认价格路径和 normal 分类） |
| `ReportService` | Agent 模拟服务失败 | 使用 Mock 推理材料（基本面/技术面分析结论） |
| `ReportService` | LLM API 调用失败 | 使用模板生成 Mock 报告文本 |
| `PredictService` | Redis 不可用 | 缓存写入失败仅记录日志，不中断预测流程 |

上述**尽力而为（Best Effort）**的多层降级机制，保证了系统在部分外部依赖不可用时仍能维持核心服务的正常响应，具备较强的生产环境鲁棒性。

## 15. ML 预测服务技术文档（model-remote 分支）

> 代码位置：仓库 `model-remote` 分支，独立 Python 项目，运行在端口 **8000**。

### 15.1 概述

ML 预测服务是基于 **FastAPI + Uvicorn** 构建的 Python 微服务，对外提供两个核心推理能力：
- **`/sentiment/score`** —— 对市场新闻进行情感评分与极端事件检测
- **`/predict`** —— 接收历史价格序列，输出多步价格预测路径及置信区间

### 15.2 服务文件结构

```
app/
├── main.py                    # FastAPI 服务入口，路由定义
├── pipeline.py                # 预测流程编排（调用各 Predictor）
├── schemas.py                 # Pydantic 请求/响应模型定义
├── config.py                  # 配置（MODEL_DIR 路径、DeepSeek API Key）
├── features.py                # 特征工程（价格变化率、技术指标拼接）
├── utils.py                   # 工具函数
└── models/
    ├── ensemble_predictor.py  # 融合预测器（AlphaPredictor + TFTCsvPredictor + MWUM）
    ├── alpha_predictor.py     # AlphaModelFrame 集成学习预测器
    ├── tft_csv_predictor.py   # TFT 预计算 CSV 查表预测器
    ├── tft_predictor.py       # TFT 原始实现（降级备用）
    ├── lgbm_predictor.py      # LightGBM 预测器（降级备用）
    └── ceemdan.py             # CEEMDAN 信号分解（趋势提取）

artifacts/                     # 预训练模型文件（不纳入 Git）
├── lgbm_model.txt
└── tft_ckpt.pt
requirements.txt               # Python 依赖
run.sh                         # 启动脚本
```

### 15.3 模型架构

服务启动时优先初始化 **EnsemblePredictor**；若模型文件缺失或加载异常，自动降级至 **StubPredictor**（基于末值线性外推的保底预测，确保接口可用性）。

```
EnsemblePredictor
├── AlphaPredictor
│   └── 加载 ~/model/AlphaModelFrame/.../model_de_*.pkl
│       → 基于 175 维特征的梯度提升集成模型
│       → 预测标的资产日收益率
│
├── TFTCsvPredictor
│   └── 读取 ~/model/TFT/gbo_tft_test_predictions_cl0_1d_close_smooth_latest.csv
│       → 时序深度学习模型的预计算预测结果（避免推理延迟）
│       → 输出带置信区间的价格路径
│
└── MWUM（Multiplicative Weights Update Method）
    └── 读取 ~/model/ensemble/output/ensemble.csv
        → 根据历史预测误差动态调整 Alpha / TFT 权重
        → eta（学习率）自适应调节，跟踪市场状态变化
```

**模型文件路径** 通过 `MODEL_DIR` 环境变量注入（默认 `~/model/`），不纳入 Git 版本管理。

### 15.4 API 接口

#### `POST /predict`

**请求体（关键字段）：**

```json
{
  "target": "WTI",
  "horizon": "1d",
  "price_series": [78.5, 79.1, ...],      // 历史收盘价序列（至少 60 条）
  "indicators": { "rsi": 62.3, ... },     // 技术指标特征
  "events": [{ "type": "GeoPolitical", "intensity": 0.8 }],
  "sentiment_score": 0.3
}
```

**响应体（关键字段）：**

```json
{
  "forecast": {
    "point": [{"t": "2025-03-01", "v": 80.2}, ...],    // 点预测路径
    "lower": [...],                                       // 90% 置信区间下界
    "upper": [...]                                        // 90% 置信区间上界
  },
  "extremeClass": { "label": "MID_RISK", "prob": 0.45 },
  "explain": {
    "feature_importance": { "price_change_1d": 0.23, ... },
    "ensemble_weights": { "alpha": 0.62, "tft": 0.38 }
  }
}
```

**多步预测原理：**
- 各预测步以当步预测收益率按复利公式逐步递推，构建完整价格路径
- 置信区间宽度随预测步数按 `sqrt(steps)` 比例扩展，体现时间维度的不确定性累积
- 输入序列中存在 `None` 值时，对应步骤自动跳过并保持前值，具备基本容错能力

#### `POST /sentiment/score`

**请求体：** `{ title, content, source, url, publish_date }`

**响应体：**

```json
{
  "sentiment": 0.72,                      // [-1, 1]，负=偏空，正=偏多
  "confidence": 0.85,
  "extreme_events": [
    { "type": "SupplyCut", "summary": "...", "intensity": 0.9 }
  ]
}
```

**实现策略：**
1. 优先调用 **DeepSeek API**（需配置 `DEEPSEEK_API_KEY` 环境变量）进行深度语义分析
2. API 不可用时自动降级为**关键词规则评分**（以 OPEC 减产、地缘冲突等供给冲击词汇为正向特征，库存超预期增加、需求萎缩等词汇为负向特征）

## 16. Agent 模拟服务技术文档（agent-service）

> 代码位置：主仓库 `agent-service/src/stagex2/` 目录，运行在端口 **8001**。

### 16.1 概述

Agent 模拟服务基于 **stagex2 框架**构建，将原油期货市场中的典型参与者（投机者、套期保值者、做市商等）建模为具有独立决策机制的智能体，模拟其在接收市场信号时的感知、决策与执行全过程。服务以 FastAPI 部署，代码已纳入主仓库统一版本管理。

### 16.2 服务文件结构

```
agent-service/src/stagex2/
├── api.py           # FastAPI 路由（单步/多步模拟 + 清算校准）
├── simulator.py     # 模拟执行器（run_single_step / run_multi_step）
├── schemas.py       # Pydantic 数据模型（市场环境/智能体状态/订单等）
├── perception.py    # 感知模块（解析市场因子、识别信号强度）
├── decision.py      # 决策模块（基于角色策略生成交易意图）
├── execution.py     # 执行模块（将意图转化为标准订单）
└── clearing.py      # 清算模块（撮合订单、更新持仓、计算盈亏）
```

### 16.3 API 接口

#### `POST /simulate/single` —— 单步模拟

输入当前市场环境快照和智能体状态，输出一个决策周期内智能体的推理过程和行为结果。

**请求关键字段：**

```json
{
  "input": {
    "environment": {
      "factor_snapshot": {
        "trend_score": 0.72,         // 趋势强度（-1 到 +1）
        "rsi_status": "Overbought",  // RSI 状态
        "term_structure": "Backwardation"  // 期限结构
      },
      "tail_risk_report": {
        "gamma_profile": "Short Gamma",
        "liquidity_stress": 0.65     // 流动性压力（0~1）
      },
      "market_microstructure": {
        "bid_ask_spread": 0.08,
        "order_book_depth": "Thin"
      },
      "session_info": { "phase": "Mid-Day", "time_to_close": 120 }
    },
    "event": {
      "headline": "OPEC hints at deeper supply cuts",
      "impact_type": "Supply",
      "sentiment_score": 1.8
    },
    "self": {
      "role": "speculator",
      "max_leverage": 3.0,
      "position": 10,
      "cash_level": 100000
    }
  },
  "current_price": 78.5,
  "current_volatility": 0.24,
  "dealer_inventory": -120
}
```

**响应：** 包含智能体的推理链、决策方向、生成的订单、以及清算后的持仓变化。

#### `POST /simulate/multi` —— 多步模拟

连续模拟多个决策周期（默认 10 步），支持智能体在时序演化中的状态传递，输出完整的决策轨迹。

#### `POST /calibrate/clearing` —— 清算参数校准

根据历史观测数据校准市场清算参数（滑点、冲击成本等），提升模拟的市场真实性。

### 16.4 智能体角色

| 角色 | 策略类型 | 决策特征 |
|------|---------|---------|
| `speculator` | 短期方向性交易 | 基于动量信号高频进出，允许较高杠杆倍数 |
| `CTA` | 系统性趋势追随 | 中期趋势驱动建仓，执行严格止损纪律 |
| `hedger` | 套期保值 | 对冲现货头寸敞口，目标维持 Delta 中性 |
| `market_maker` | 做市商 | 持续提供双边报价流动性，主动管理库存风险 |

### 16.5 与 Java 后端的集成

Java 端通过 `Stage2AgentClient` 调用 Agent 服务：

```java
// AgentService.java
Stage2SingleRequest request = buildDemoRequest();  // 构造市场环境 + 智能体状态
JsonNode result = stage2AgentClient.simulateSingle(request);
// result 包含决策推理材料，用于后续 LLM 报告生成

// Stage2AgentClient.java
POST http://<stage2.base-url>/simulate/single
Content-Type: application/json
```

`stage2.base-url` 通过 `application.yml` 配置（默认 `http://localhost:8001`）。

**联调验证端点：** `GET /api/agent/demo/simulate-single` 使用预置的标准场景参数（OPEC 供应冲击事件 + 投机者角色）直接调用 Agent 服务并返回完整模拟结果，可用于前后端集成验证及功能展示。
