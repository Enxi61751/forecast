## 1. 项目概述

本系统是面向金融机构的**原油价格智能预测与风险分析平台**后端服务，旨在为银行大宗商品交易团队提供数据驱动的油价预测、舆情感知和风险评估能力。系统整合了多源数据（实时油价、宏观经济指标、新闻舆情、地缘政治事件），借助机器学习预测模型与多智能体博弈模拟，输出结构化预测结果和可读性强的风险报告，辅助交易员和风控人员的决策制定。

**核心能力：**

- **实时油价采集** —— 自动拉取 WTI/Brent 原油实时及历史行情（OHLCV）
- **新闻舆情分析** —— 接入新闻稿件，调用 NLP 模型进行情感评分与极端事件检测
- **多模型融合预测** —— 融合 AlphaModelFrame 集成学习、TFT（Temporal Fusion Transformer）深度学习、MWUM 动态加权三类模型，输出多时间尺度价格预测
- **多智能体风险模拟** —— 基于 stagex2 框架，模拟 CTA、套期保值者等不同市场参与者的博弈行为
- **LLM 报告生成** —— 调用 DeepSeek 大语言模型，将量化分析材料转化为面向业务人员的自然语言风险报告

## 3. 系统架构设计

本系统采用**经典分层架构**，并通过微服务解耦将机器学习推理与业务逻辑分离：

│  │ Controller │──▶│  Service  │──▶│ Repository │──▶│  MySQL DB    │ │
│  │   Layer    │   │   Layer   │   │   Layer    │   │              │ │
│  └───────────┘   └─────┬─────┘   └────────────┘   └──────────────┘ │

└───────────────────┼────┼───┼────────────────────────────────────────┘
                    │    │   │
         ┌──────────┘    │   └──────────┐
         ▼               ▼              ▼
┌────────────────┐ ┌──────────┐ ┌──────────────────┐
│ ML Model 服务   │ │ Agent    │ │ DeepSeek LLM API │

**架构要点：**

1. **关注点分离** —— Controller 仅负责请求路由与参数校验，Service 封装业务逻辑，Repository 处理数据持久化
2. **模型服务解耦** —— ML 推理部署为独立的 Python FastAPI 服务，通过 HTTP/JSON 与 Java 后端通信，避免语言绑定
3. **弹性降级** —— 每个外部服务调用均有 try-catch 包裹，失败时自动回退到 Mock 数据，保证系统可用性
4. **缓存加速** —— Redis 缓存最新预测结果，减少重复计算开销

## 6. 核心功能模块详解

### 6.1 原油价格数据采集

**涉及组件：** `OilPriceController` → `OilPriceService` → `OilPriceHistoryRepo` / 外部 API

**实现原理：**

系统通过两种途径获取油价数据：

1. **实时拉取（按需）** —— 前端调用 `GET /api/oilprice/latest` 时，`OilPriceService` 通过 `RestTemplate` 向 OilPriceAPI（`https://api.oilpriceapi.com/v1/prices/latest`）发起带 Bearer Token 的 HTTP GET 请求，获取 WTI 和 Brent 两个品种的最新价格，返回标准化的 `OilPriceApiResponse` 给前端。

2. **定时采集（每日）** —— `DailyPipelineJob` 以 cron 表达式 `0 0 1 * * ?`（每日凌晨 1:00）触发，调用 `OilPriceService.fetchAndSaveOilPrice()`，将 WTI 和 Brent 的最新价格分别持久化为 `OilPriceHistory` 实体存入数据库。价格时间戳通过解析 API 返回的 ISO 8601 `OffsetDateTime` 并转换为 `LocalDateTime` 实现时区对齐。

3. **历史查询** —— `GET /api/oilprice/history?start=...&end=...` 支持按时间范围查询数据库中的油价历史记录，利用 JPA 的 `findByTimestampBetween` 自动生成 SQL Range 查询。

**容错设计：** 若 API Token 未配置（保持默认值 `YOUR_OIL_API_TOKEN`）或 API 调用失败，自动返回 Mock 数据，避免服务不可用。

### 6.2 新闻舆情分析与情感评分

**涉及组件：** `NewsController` → `NewsService` → `ModelServiceClient` → ML 模型服务（port 8000）

**实现原理：**

新闻舆情分析采用**采集-评分-存储**三阶段流水线设计：

```
POST /api/news/ingest
        │
        ▼
┌─────────────────────────┐
│ Step 1: 持久化原始稿件    │  @Transactional
│ NewsArticle → MySQL     │  ← 独立事务，确保稿件不丢失
└────────────┬────────────┘
             │ newsId
             ▼
┌─────────────────────────┐
│ Step 2: 调用 NLP 模型    │  HTTP POST → :8000/sentiment/score
│ 情感评分 + 极端事件检测   │  ← 事务外执行，失败不影响 Step 1
└────────────┬────────────┘
             │ SentimentResponse
             ▼
┌─────────────────────────┐
│ Step 3: 持久化评分结果    │  @Transactional
│ SentimentScore + Event  │  ← 评分与事件原子写入
└─────────────────────────┘
```

**关键设计决策：**

- **事务边界精细控制** —— `saveArticle()` 和 `saveSentiment()` 各自拥有独立的 `@Transactional` 注解。模型调用（Step 2）位于两个事务之间，确保即使 NLP 服务不可用，原始新闻稿件依然完整保存。
- **极端事件检测** —— 模型服务返回的 `SentimentResponse` 中可包含 `extremeEvents` 列表（如地缘政治冲击 GeoPoliticalShock、OPEC 减产 SupplyCut 等），每个事件带有类型、摘要、强度和发生时间，系统逐一写入 `ExtremeEvent` 表。
- **异步降级** —— 情感评分失败仅记录警告日志，不影响新闻入库的返回，前端可后续重试评分。

### 6.3 多模型融合预测引擎

**涉及组件：** `PredictController` → `PredictService` → `ModelServiceClient` → ML 模型服务（port 8000）→ Redis

**实现原理：**

预测引擎是系统的核心功能，整合了三类机器学习模型实现多时间尺度的油价预测：

#### 模型架构

| 模型 | 类型 | 职责 |
|------|------|------|
| **AlphaModelFrame** | 集成学习（pkl 模型文件） | 基于 175 维特征数据集的收益率预测，使用预训练的集成模型或预计算的预测结果 |
| **TFT (Temporal Fusion Transformer)** | 深度学习时序模型 | 油价时序预测，输出带置信区间的价格路径，同时提供极端分类能力 |
| **MWUM (Multiplicative Weights Update Method)** | 动态权重融合 | 基于各子模型历史表现动态调整融合权重，自适应市场状态变化 |

#### 预测请求处理流程

```
前端 POST /api/predict
  Body: { target: "WTI", horizon: "7d", payload: {...} }
         │
         ▼
┌─────────────────────────────────────────┐
│ ModelServiceClient.predict()            │
│ 将 payload 展平，与 target/horizon 合并 │
│ POST → :8000/predict                    │
└────────────────┬────────────────────────┘
                 │ PredictResponse { forecast, extremeClass, explain }
                 ▼
┌─────────────────────────────────────────┐
│ 持久化到 prediction_run 表              │
│ forecast → forecastJson (JSON TEXT)     │
│ extremeClass → extremeClsJson          │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│ Redis 缓存                              │
│ Key: pred:latest:{target}:{horizon}     │
│ Value: predictionRunId                  │
│ TTL: 5 分钟                             │
└─────────────────────────────────────────┘
```

**payload 灵活设计** —— `PredictRequest.payload` 采用 `Map<String, Object>` 类型，允许前端传入任意结构的附加参数（如特征数据、时间窗口约束等），`ModelServiceClient` 在发送前将 payload 与 target/horizon 合并为扁平化请求体，适配 Python 模型服务的接口格式。

**预测结果结构** —— `PredictResponse` 包含三个维度：
- `forecast`：价格预测结果（可为路径、区间或点预测）
- `extremeClass`：极端行情分类结果（如 normal/spike/crash 及置信度）
- `explain`：模型可解释性信息（特征重要性等）

### 6.4 多智能体模拟与风险报告生成

**涉及组件：** `ReportController` → `ReportService` → `ModelServiceClient` → Agent 服务（port 8001）+ DeepSeek LLM API

**实现原理：**

风险报告生成是系统最具创新性的功能，采用**"多智能体博弈模拟 + 大语言模型文本生成"**的双阶段架构：

#### 阶段一：多智能体博弈模拟（stagex2 框架）

系统基于 stagex2 框架构建市场微观结构模拟环境，将不同类型的市场参与者建模为智能体，模拟其面对油价预测信号时的行为决策：

```json
{
  "input": {
    "environment": {
      "factor_snapshot": {           // 因子快照
        "trend_score": 0.0,          // 趋势得分
        "rsi_status": "Neutral",     // RSI 状态
        "term_structure": "Contango" // 期限结构
      },
      "tail_risk_report": {          // 尾部风险报告
        "gamma_profile": "Neutral",  // Gamma 暴露
        "liquidity_stress": 0.3      // 流动性压力
      },
      "market_microstructure": {     // 市场微观结构
        "bid_ask_spread": 0.05,      // 买卖价差
        "order_book_depth": "Normal" // 订单簿深度
      },
      "session_info": {              // 交易时段信息
        "phase": "Mid-Day",
        "time_to_close": 240         // 距收盘分钟数
      }
    },
    "event": {                       // 触发事件（来自预测结果）
      "headline": "Oil price forecast: WTI over 7d",
      "body": "<forecastJson>",
      "source": "CitiCup Model",
      "impact_type": "price",
      "sentiment_score": 0.0
    },
    "self": {                        // 默认 CTA 智能体状态
      "role": "CTA",
      "mandate": "trend-following momentum strategy",
      "max_leverage": 3.0,
      "stop_loss_pct": 0.05,
      "position": 0.0,
      "cash_level": 1000000.0
    }
  },
  "current_price": 80.0,
  "current_volatility": 0.2,
  "dealer_inventory": 100
}
```

Agent 服务返回各智能体的决策推理材料（`materialsJson`），为后续报告撰写提供量化论据。

#### 阶段二：LLM 报告生成

系统将智能体模拟产出的结构化材料作为上下文，通过 `ModelServiceClient.callLlm()` 调用 DeepSeek Reasoner 大语言模型（`deepseek-reasoner`），生成面向银行业务团队的自然语言风险报告。

LLM 调用采用标准 OpenAI 兼容接口：
- 请求格式：`{"model": "deepseek-reasoner", "messages": [{"role": "user", "content": prompt}]}`
- 认证方式：`Authorization: Bearer <api-key>`
- 响应解析：提取 `choices[0].message.content`

#### Upsert 语义保证

每个 `PredictionRun` 仅对应一份 `RiskReport`（数据库唯一约束 `unique_prediction`）。若对同一预测重复生成报告，系统查找已有记录并执行更新（而非插入），避免数据冗余。

## 7. 外部服务集成

系统通过 `ModelServiceClient` 统一管理所有外部服务调用，集成了三个独立端点：

| 服务 | 地址 | 协议 | 功能 |
|------|------|------|------|
| **ML 模型服务** | `http://localhost:8000` | REST/JSON | `/predict`（油价预测）、`/sentiment/score`（情感评分） |
| **Agent 模拟服务** | `http://localhost:8001` | REST/JSON | `/simulate/single`（单步智能体博弈模拟） |
| **DeepSeek LLM** | `https://api.deepseek.com/chat/completions` | OpenAI 兼容 | 自然语言报告生成 |

**ML 模型服务内部架构（FastAPI，model-remote 分支）：**

```
FastAPI :8000
├── /predict          → pipeline.py
│   ├── EnsemblePredictor (优先)
│   │   ├── AlphaPredictor   ← ~/model/AlphaModelFrame/.../model_de_*.pkl
│   │   ├── TFTCsvPredictor  ← ~/model/TFT/..._predictions_*.csv
│   │   └── MWUM 权重       ← ~/model/ensemble/output/ensemble.csv
│   └── StubPredictor (降级)
│
└── /sentiment/score  → 情感分析模型
```

模型文件存储在服务器本地 `~/model` 目录下，通过 `MODEL_DIR` 环境变量配置路径，不纳入 Git 版本管理。

## 8. 缓存机制

系统使用 Redis 作为预测结果的热缓存层，避免短时间内对同一标的和时间窗口的重复预测请求触发昂贵的模型推理。

**缓存策略：**

| 属性 | 值 |
|------|------|
| 存储介质 | Redis（localhost:6379） |
| 客户端 | Spring `StringRedisTemplate` |
| Key 格式 | `pred:latest:{target}:{horizon}`（如 `pred:latest:WTI:7d`） |
| Value | `PredictionRun` 主键 ID |
| TTL | 5 分钟（`Duration.ofMinutes(5)`） |
| 失败处理 | 缓存写入失败仅记录警告日志，不影响预测结果返回 |

**设计考量：** 缓存中仅存储预测记录 ID 而非完整预测结果，避免大 JSON 对象占用过多 Redis 内存，需要完整数据时可通过 ID 回查 MySQL。

## 9. 定时任务与数据管道

### 9.1 每日油价采集任务

```java
@Scheduled(cron = "0 0 1 * * ?")  // 每日 01:00 执行
public void fetchOilPrice() {
    oilPriceService.fetchAndSaveOilPrice();
}
```

`DailyPipelineJob` 利用 Spring 的 `@Scheduled` 注解实现 cron 定时触发，配合主类上的 `@EnableScheduling` 注解生效。任务执行时调用外部油价 API 并将 WTI/Brent 两个品种的最新价格分别写入数据库。

### 9.2 应用启动数据初始化

`DataInitializer` 实现 `CommandLineRunner` 接口，在 Spring Boot 完成上下文初始化后自动执行：

1. 检查数据库中是否已存在新闻数据
2. 若为空库，初始化基础演示数据：3 条新闻、6 条汇率、16 条油价历史（8 天 × 2 品种）、3 次预测记录、1 份风险报告
3. 独立检查并初始化事件数据（3 条地缘/经济事件）

该机制确保系统在首次部署或数据库重建后即可进入可演示状态。

## 10. 全局异常处理与统一响应格式

### 10.1 统一响应封装

所有 API 接口均返回 `ApiResponse<T>` 格式，确保前端解析逻辑统一：

```json
// 成功
{ "code": 0, "message": "ok", "data": { ... } }

// 失败
{ "code": -1, "message": "错误描述", "data": null }
```

### 10.2 全局异常拦截

`GlobalExceptionHandler`（`@RestControllerAdvice`）拦截所有未捕获异常并转换为标准响应格式：

| 异常类型 | HTTP 状态码 | 处理方式 |
|---------|-------------|---------|
| `MethodArgumentNotValidException` | 400 | 提取字段校验错误信息 |
| `ConstraintViolationException` | 400 | 提取约束违反信息 |
| `IllegalArgumentException` | 400 | 返回参数错误描述 |
| `Exception`（兜底） | 500 | 返回通用错误信息 |

## 12. 容错与降级策略

系统在多个层面实现了弹性容错设计，确保外部服务故障不会导致整体不可用：

| 组件 | 容错场景 | 降级策略 |
|------|---------|---------|
| `OilPriceService` | API Token 未配置 / API 调用失败 | 返回 Mock 油价数据（WTI $75.50, Brent $80.25） |
| `NewsService` | NLP 模型服务不可达 | 新闻稿件正常入库，情感评分跳过，记录警告日志 |
| `PredictService` | 预测模型服务调用失败 | 返回 Mock 预测结果（含默认价格路径和 normal 分类） |
| `ReportService` | Agent 模拟服务失败 | 使用 Mock 推理材料（基本面/技术面分析结论） |
| `ReportService` | LLM API 调用失败 | 使用模板生成 Mock 报告文本 |
| `PredictService` | Redis 不可用 | 缓存写入失败仅记录日志，不中断预测流程 |

这种**"尽力而为"（Best Effort）**的降级策略保证了系统在部分依赖不可用时仍能提供基础功能。

## 14. 部署架构

系统由四个独立进程协作运行：

```
启动顺序：
1. MySQL + Redis          ← 基础设施（预先部署）
2. ./start-services.sh    ← 启动 ML 模型服务 (:8000) + Agent 服务 (:8001)
3. ./start.sh             ← 启动 Spring Boot 后端 (:8080)
4. 前端 dev server         ← Vue 3 开发服务器 (:5173)
```

| 进程 | 端口 | 技术栈 | 说明 |
|------|------|--------|------|
| Spring Boot 后端 | 8080 | Java 17 + Spring Boot 3.3.2 | 业务编排、数据持久化、API 网关 |
| ML 模型服务 | 8000 | Python + FastAPI | 油价预测（Alpha + TFT + MWUM 融合）、情感评分 |
| Agent 模拟服务 | 8001 | Python + stagex2 | 多智能体市场博弈模拟 |
| MySQL 数据库 | 3306 | MySQL 8.x | 关系型数据持久化（远程部署） |
| Redis | 6379 | Redis | 预测结果热缓存（本地部署） |
| 前端 | 5173 | Vue 3 + Vite | 可视化交互界面 |

**模型文件管理：** 训练好的模型文件（pkl、csv）存储在服务器本地 `~/model` 目录，通过 `MODEL_DIR` 环境变量注入 FastAPI 服务，不纳入 Git 版本管理，确保仓库体积可控。
